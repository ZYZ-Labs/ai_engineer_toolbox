# RAG Roadmap

## Goal

RAG is an information system before it is an LLM pattern. A reliable RAG system needs document modeling, retrieval quality, evaluation data, prompt discipline, and observability.

## Learning Path

1. Map the source corpus. Identify document types, update cadence, access control, and fields that must survive ingestion.
2. Design chunks around user questions, not arbitrary character counts. Add stable IDs, section titles, source URLs, and timestamps.
3. Build retrieval baselines. Compare keyword, vector, and hybrid retrieval before optimizing generation.
4. Create an evaluation set. Include answerable, unanswerable, ambiguous, stale, and permission-sensitive questions.
5. Add reranking and answer synthesis only after retrieval metrics expose the current bottleneck.
6. Instrument the full path: query rewrite, retrieved chunks, scores, selected context, model settings, answer, and citations.

## Engineering Practices

- Store ingestion versions so you can reproduce an answer later.
- Evaluate retrieval and generation separately.
- Keep chunk metadata visible in debugging tools.
- Add a refusal path for missing evidence.
- Test stale-document behavior before production users find it.

## Pitfalls

- Treating embedding search as a database replacement.
- Hiding retrieved context from logs and then being unable to debug answers.
- Mixing private and public documents without a permissions model.
- Measuring only answer fluency instead of source grounding.

## Recommended Tools

- Use JSON Formatter for retrieval traces and API responses.
- Use JSON Diff to compare index metadata before and after ingestion changes.
- Use URL Encoder when testing search and document source links.

## Checkpoint

You should be able to answer: Which document supported this answer, why was it retrieved, what version was indexed, and what happens when no reliable evidence exists?
